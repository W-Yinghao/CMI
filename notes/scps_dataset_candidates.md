# RECOMMENDATION (the answer to "multi-level D + SCPS focus")

**The registry resolves the earlier impasse.** The clean way to get a *real* multi-level D — without
re-preprocessing TUAB (v3.0.1 is single-montage `01_tcp_ar`) — is **cross-dataset SCPS (Protocol C) with
hierarchical `D = (cohort/site, subject)`**. Same-disease cohorts come from *different sites/devices*, so D
genuinely has two informative levels and `π_y` becomes a hierarchical prior `π(cohort)·π(subject|cohort)` =
the new method. **Almost all candidates are OpenNeuro (BIDS, open, no agreements)** → buildable now.

## Flagship: PARKINSON (8+ OpenNeuro PD-vs-HC, several device-matched)
Use the **resting-state** portions; binary PD/HC; canonicalize to 19-ch 10-20; train on K−1 cohorts, test on
held-out cohort (cross-site+device+subject).

- **Schizophrenia (bonus):** 3× First-Episode-Psychosis **ds003947/ds003944** + **ds004000** + **ds004367** + Zenodo GM (29601).

## Why this is the right move
1. **Makes π_y multi-level** (cohort + subject) — a genuine methodological extension, not a re-run.
2. **Real cross-site/cross-device clinical DG** = the actual deployment problem (train hospital A → deploy B). High impact.
3. **Keeps SCPS focus** (where π_y is identifiable) — fixes the "π_y invisible on MCPS" reviewer worry.
4. **No TUAB-raw / no data agreements** — OpenNeuro BIDS, our pipeline already does 19-ch canonicalization + resample.

## Honest costs / risks
- **Task heterogeneity:** PD cohorts are different *tasks* (rest/oddball/Simon/RL) → use **resting-state only** (most have it) to avoid a task confound. Same for AD/Dep.
- **Acquisition harmonization:** channels 32–256, SR 250–2048, different amplifiers → 19-ch 10-20 intersection + resample (pipeline exists); but this *is* the leakage we study, so keep it (don't over-normalize).
- **Engineering:** generic BIDS-EEG loader + per-cohort label map + download ~tens of GB. ~1–2 weeks. Payoff: the multi-level-D contribution + a real cross-site clinical benchmark.

## Direction 1 (TUAB montage-D) — DOWNGRADED
TUAB v3.0.1 on disk is single-montage (`01_tcp_ar`); ar/le/ar_a is gone. The full **TUEG/TUSZ/TUEP** corpora
(in registry, N=10,874+) retain montage/era diversity but are a much larger undertaking. Not the efficient path.
